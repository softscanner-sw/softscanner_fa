# B0 Manifest & CLI Guide

**Purpose:** This guide explains how to create, validate, and maintain subject manifests for the SoftScanner FA benchmark pipeline.

---

## What is the manifest?

The **SubjectManifest** (`subjects/<name>/subject-manifest.json`) is the canonical contract between the pipeline and the running application. It declares everything the pipeline needs to generate and execute tests: authentication, route parameters, form data, execution configuration, and seed requirements.

The manifest is:
- **Generated by the wizard** (`npm run b0:wizard`)
- **Validated by B0** (`npm run b0:validate`)
- **Read-only at runtime** — never mutated by the pipeline

---

## Quick start

```bash
# 1. Generate a manifest (interactive wizard)
npm run b0:wizard

# 2. Validate it
npm run b0:validate

# 3. Generate tests
npm run b2:codegen

# 4. Run benchmark
node node_modules/tsx/dist/cli.mjs src/b3-cli.ts <subjectName> --max-retries 1 --batch-size 10
```

## What the wizard infers

The wizard reads A2 workflow artifacts and infers:

| Requirement | Source | What you must provide |
|---|---|---|
| Auth guards needed | A2 `cw.guards` | Credentials, roles, guard satisfaction |
| Route params needed | A2 `cw.requiredParams` | Concrete entity IDs |
| Form workflows present | A2 trigger edge kinds | Nothing (informational) |
| Auth setup mechanism | Guard presence | Login route, form selectors, success selector |

The wizard **cannot** infer:
- Backend-valid concrete values (credentials, UUIDs, entity IDs)
- Seed data commands (how to populate the database)
- Rate-limit reset commands
- Timeout profiles

These must be provided by you during the wizard or added to the manifest afterward.

## Worked examples

### Example A: Simple no-auth subject (heroes-angular)

No guards, no route params, no forms. The simplest case.

```json
{
  "subjectName": "heroes-angular",
  "baseUrl": "http://localhost:7626",
  "accounts": [],
  "routeParamValues": {},
  "seedRequirements": {
    "authGuards": [],
    "negativeGuards": [],
    "routeParams": [],
    "hasFormWorkflows": false,
    "seedStatus": "none"
  }
}
```

**Why:** No auth → no accounts needed. No `:id` params → no entity IDs needed. Data comes from a static `db.json` file.

### Example B: Auth-protected subject (airbus-inventory)

One guard (`CanActivateRouteGuard`), no route params, form submissions.

```json
{
  "subjectName": "airbus-inventory",
  "baseUrl": "http://localhost:4200",
  "accounts": [
    {
      "username": "airbus02@gmail.com",
      "password": "1234",
      "roles": ["user"],
      "guardSatisfies": ["CanActivateRouteGuard"]
    }
  ],
  "authSetup": {
    "loginRoute": "/login",
    "usernameField": "input[formcontrolname='emailid']",
    "passwordField": "input[formcontrolname='password']",
    "submitButton": "button[type='submit']",
    "authSuccessSelector": "mat-sidenav-container"
  },
  "seedRequirements": {
    "authGuards": ["CanActivateRouteGuard"],
    "negativeGuards": [],
    "routeParams": [],
    "hasFormWorkflows": true,
    "seedStatus": "needs-command"
  }
}
```

**Why `authSuccessSelector`:** After login, the pipeline waits for `mat-sidenav-container` to appear. This is the post-auth UI element that proves login succeeded.

**Why `seedStatus: "needs-command"`:** The backend needs a MySQL database with the User table seeded. Without it, login fails.

### Example C: Route-param-heavy subject (spring-petclinic-angular)

No auth, but many `:id` route parameters across different entity families.

```json
{
  "subjectName": "spring-petclinic-angular",
  "baseUrl": "http://localhost:4200",
  "accounts": [],
  "routeParamValues": { "id": "1" },
  "routeParamOverrides": {
    "/owners/:id": { "id": "1" },
    "/owners/:id/edit": { "id": "1" },
    "/owners/:id/pets/add": { "id": "1" },
    "/pets/:id/edit": { "id": "1" },
    "/pets/:id/visits/add": { "id": "1" },
    "/pettypes/:id": { "id": "1" },
    "/specialties/:id": { "id": "1" },
    "/vets/:id/edit": { "id": "1" },
    "/visits/:id/edit": { "id": "1" }
  },
  "seedRequirements": {
    "authGuards": [],
    "negativeGuards": [],
    "routeParams": ["id"],
    "hasFormWorkflows": true,
    "seedStatus": "pre-seeded"
  }
}
```

**Why `routeParamOverrides`:** All entity families use `:id` but refer to different entity types. The overrides ensure the correct entity is referenced per route.

**Why `id: "1"`:** The Docker image comes pre-seeded with entities starting at id=1.

### Example D: Backend-validation-sensitive + batch-reset subject (ever-traduora)

Multiple guards, route params with dynamic UUIDs, auth rate limiting.

```json
{
  "subjectName": "ever-traduora",
  "baseUrl": "http://localhost:4200",
  "accounts": [
    {
      "username": "admin@test.com",
      "password": "Test1234!",
      "roles": ["admin"],
      "guardSatisfies": ["AuthGuard", "CanGuard"]
    },
    {
      "username": "user@test.com",
      "password": "Test1234!",
      "roles": ["user"],
      "guardSatisfies": ["AuthGuard"]
    }
  ],
  "routeParamValues": {
    "localeCode": "en",
    "projectId": "<UUID from seed>"
  },
  "authSetup": {
    "loginRoute": "/login",
    "usernameField": "input[formcontrolname='email']",
    "passwordField": "input[formcontrolname='password']",
    "submitButton": "button[type='submit']",
    "authSuccessSelector": "app-bar [routerLink='/projects']"
  },
  "executionConfig": {
    "preAttemptCommand": "docker exec mysqldb mysql -u tr -pchange_me tr_dev -e \"UPDATE user SET loginAttempts = 0, lastLogin = NULL;\"",
    "batchResetCommand": "docker restart traduora && sleep 10",
    "enableNetworkEvidence": true
  },
  "seedRequirements": {
    "authGuards": ["AuthGuard", "CanGuard"],
    "negativeGuards": ["NoAuthGuard"],
    "routeParams": ["localeCode", "projectId"],
    "hasFormWorkflows": true,
    "seedStatus": "needs-command"
  }
}
```

**Why `preAttemptCommand`:** Resets MySQL login attempt counters before each test (prevents DB-level lockout).

**Why `batchResetCommand`:** Restarts the traduora Docker container every 10 tests to clear the in-memory NestJS rate limiter.

**Why `projectId: "<UUID from seed>"`:** The projectId is generated dynamically by the seed script. After seeding, you must update this value in the manifest.

## Common mistakes

| Mistake | Symptom | Fix |
|---|---|---|
| Wrong `baseUrl` | FAIL_APP_NOT_READY or wrong app | Verify the correct port per runbook |
| Missing `authSuccessSelector` | FAIL_TIMEOUT at auth step | Add the CSS selector for post-login element |
| Missing `routeParamValues` | `undefined` in URLs, HTTP 400 | Add concrete values from seeded data |
| No `preAttemptCommand` for rate-limited apps | HTTP 429 after ~80 tests | Add DB reset or rate-limit reset command |
| No `batchResetCommand` for in-memory rate limiters | HTTP 429 mid-run | Add Docker restart command |
| Stale `routeParamValues` after re-seeding | Tests fail on entity-specific routes | Re-run wizard or update values manually |

## `seedRequirements` field

This field is **generated automatically** by the wizard from A2 artifacts. You should NOT edit it manually.

| Field | Meaning |
|---|---|
| `authGuards` | Positive auth guards requiring accounts (e.g., AuthGuard, CanGuard) |
| `negativeGuards` | Negative guards requiring NO auth (e.g., NoAuthGuard — login/signup pages) |
| `routeParams` | Route params that require entity values |
| `hasFormWorkflows` | Whether any workflow submits a form (backend validation relevant) |
| `seedStatus` | `"none"` (no seed needed), `"pre-seeded"` (Docker/static seed), or `"needs-command"` (requires seedCommand) |

B0 validates `seedRequirements` against the rest of the manifest:
- If an auth guard is listed but no account satisfies it → **warning**
- If a param is listed but no value is provided → **warning**
- If `seedStatus` is `"needs-command"` but no `seedCommand` is provided → **warning**
- Negative guards (NoAuthGuard) do NOT generate warnings — they require no account

## Validation

```bash
npm run b0:validate
```

This checks:
- Schema correctness (types, required fields)
- Cross-references (guard names vs A2, param names vs A1)
- Seed requirements vs declarations
- Auth setup completeness

All manifests must be VALID before benchmark execution.
